# 0018 — SLSA L2 today, L3 plan for next half

**Status:** Accepted
**Date:** 2026-04-25
**Authors:** @ciupsciups
**Doctrine rules:** 53, 65
**Review date:** 2026-10-25 (advance to L3 evaluation point)
**Mitigates:** RISK-005, RISK-017.

## Context

SLSA (Supply-chain Levels for Software Artifacts) defines progressive supply-chain integrity targets. `slsa-github-generator/.github/workflows/generator_container_slsa3.yml` outputs L3-ready provenance, but full L3 requires hermetic builds + two-party review + external audit, which we are not yet ready to commit to operationally.

## Decision

- **Today (S3 close):** target SLSA **Level 2**. Provenance generated by `slsa-github-generator` (hosted GitHub-controlled builder); Cosign attest the predicate. Kyverno admission verifies presence of provenance.
- **Plan (next half):** target SLSA **L3**. Requires:
  - Hermetic, fully-reproducible builds (no network access during build; all deps vendored).
  - Two-party review on every workflow file change (`@greenmetrics/secops` + `@greenmetrics/platform-team` already enforced via CODEOWNERS).
  - External audit of provenance chain (annual external pentest scoped to include build chain).
- Document progress in this ADR; flip status to `Superseded by 00NN` when L3 ships.

## Alternatives considered

- **L4 (highest).** Rejected — requires offline build environment + multi-party signing; operational complexity > value for our scale + threat model.
- **L1 only.** Rejected — provides documentation but no verifiable provenance.
- **No SLSA target.** Rejected — Rule 53.

## Consequences

### Positive

- Provenance attached to every image; verifiable via `cosign verify-attestation --type slsaprovenance`.
- Kyverno admission can require provenance presence (`policies/kyverno/verify-images.yaml`).
- Aligns with NIS2 supply-chain expectation.

### Negative

- L3 migration requires `go mod vendor` + Dockerfile changes for hermetic builds — non-trivial.
- L3 review cadence increases reviewer burden on workflow PRs.

### Neutral

- L2 → L3 is incremental; no rewrite required.

## Residual risks

- `slsa-github-generator` itself is a dependency. Mitigation: pinned by SHA; review annually.
- Provenance can be verified but not independently audited until external pentest scoped (Q3 2026 external pentest).

## Tradeoff Stanza

- **Solves:** unattested artefacts; missing supply-chain integrity claim.
- **Optimises for:** progressive maturity; operational realism; verifiable per-image provenance now.
- **Sacrifices:** L3 hermetic-build complexity deferred; reliance on slsa-github-generator dependency.
- **Residual risks:** generator compromise (pin + annual review); L3 audit pending Q3 pentest scope.
