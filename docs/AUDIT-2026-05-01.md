## §1 — Security / CTF-style Audit

> **Pass owner:** Adversarial red-team review.
> **Verdict (this pass):** Would NOT pass an enterprise security audit today. "Audit-grade architecture, demo-grade implementation."

### 1.1 Attack-surface summary

The HTTP surface is centralised in `backend/internal/handlers/handlers.go`. Public unauthenticated endpoints: `/api/health`, `/api/ready`, `/api/live`, `/api/v1/auth/login`, `/api/v1/auth/refresh`, `/api/docs`, `/api/internal/metrics`. Every other route lives under `protected` group, gated by `JWTMiddleware` → `AuditMiddleware` → optional `IdempotencyMiddleware`, then a per-endpoint `security.RequirePermission(...)` check. CSRF (`security.go:92`) covers cookie-based clients; Bearer-auth bypasses (intentional, double-submit-cookie pattern). Rate limit (`security.go:152`) has three tiers: default 60/min, login `RateLimitLoginPerMinute`, ingest `RateLimitIngestPerMinute`. Defence-in-depth chain: Fiber `recover.New` → `requestid` → OTel → `compress` → `SecurityHeaders` (CSP/HSTS/X-Frame-DENY/Permissions-Policy in production) → CORS (default dev origins). Container surface: distroless non-root with digest-pinned base. K8s deployment uses `runAsNonRoot`, `allowPrivilegeEscalation: false`, `readOnlyRootFilesystem: true`. Terraform RDS has `storage_encrypted`, `publicly_accessible: false`, KMS-encrypted state with DynamoDB lock. Production boot refuses default JWT secret (`config.go:177`) and `sslmode=disable` (`config.go:184`).

### 1.3 Top 5 highest-risk items — exploit walkthroughs

#### #1 — Cross-tenant write via missing JWT claim (`readings.go:39-42`)

**Setup:** Attacker registers as a free-tier user / test tenant in a deployment with multiple tenants. Their issued JWT includes `sub`, `role`, but a bug or admin misconfiguration omits `tenant_id` (e.g. SSO bridge that doesn't map the org claim).

**Exploit:** Attacker calls `POST /api/v1/readings/ingest` with a body referencing meter_id `<victim's-meter>`. Handler reads `tenant_id` from `c.Locals("tenant_id")`, finds `nil`, falls through to `tenantID = "00000000-0000-4000-8000-000000000001"`. Inserts a row into the hypertable for that tenant. If a real customer happens to own that UUID (or even if they don't, a future onboarding will), the readings exist.

**Impact:** Data poisoning of the regulatory baseline → ESRS E1 / Piano 5.0 dossiers report wrong numbers → audit failure. Worse, because the audit middleware uses the JWT's tenant_id (also nil), the audit row attributes the action to no tenant.

**Fix:** `if tenantID == "" { return Forbidden("missing tenant claim") }`. Apply at both `Ingest` and `Aggregated`.

#### #2 — Cross-tenant DPO via DSAR URL param (`dsar.go:68-88`)

**Setup:** Attacker is a legitimate DPO at tenant A. Their JWT carries `role=dpo, tenant_id=A`.

**Exploit:** `POST /api/v1/dsar/<tenantB-uuid>/<victim-userid>/export`. RBAC sees `role=dpo`, grants `PermDSARExport`. Handler reads `tenant_id` from URL params (NOT JWT), validates them as UUIDs, returns 202 with a JobID and a `bundle_url` placeholder. If/when Phase F wires the actual export, this DPO would receive tenant B's data.

**Impact:** Today: only audit-log poisoning + fake JobID enumeration. After Phase F: full cross-tenant PII export.

**Fix:** Require `c.Params("tenant_id") == c.Locals("tenant_id")` (or document DPO-cross-tenant intent in Charter and gate via a separate `RolePlatformDPO`).

#### #3 — Auth fallback via dev sentinel (`auth.go:121-128`)

**Setup:** Engagement deploys to staging at `https://staging.client.example` with `APP_ENV=staging`. The DB pool fails to connect (network blip, accidental drop).

**Exploit:** `POST /api/v1/auth/login` with `{"email":"operator@greenmetrics.local","password":"any-12-char-string"}`. The handler sees `cfg.AppEnv != "production"`, accepts. Issues a real signed JWT for `tenantID=00000000-0000-4000-8000-000000000001, role=operator`. Attacker now has operator on the placeholder tenant.

**Impact:** Authentication bypass on any non-production environment. If the dev fallback tenant happens to have data (e.g. seed data shared with staging), the data is exposed.

**Fix:** Gate dev fallback on `strings.EqualFold(cfg.AppEnv, "development")` — exact match, not "anything but production".

#### #4 — Cosign / Pack-loader theatre

**Setup:** None — this is a doctrine-vs-code mismatch with security implications.

**Exploit:** Charter §4 + Doctrine Rules 73, 75, 124 claim Pack manifests are Cosign-verified at boot. An attacker (or supply-chain compromise) replaces a Pack file under `backend/packs/factor/ispra/factor.go` with one returning manipulated emission factors. The deployment boots normally — no signature verification. ESRS E1 dossiers are computed with attacker-controlled factors.

**Impact:** Supply-chain compromise undetected; regulatory dossiers built on poisoned inputs. The "audit-grade reproducibility" claim collapses.

**Fix:** Either implement signature verification (sigstore-go cosign verifyBlob over each Pack manifest at boot) or amend doctrine to mark this as Phase G-deferred and remove the "verified" wording from CHARTER §4.

#### #5 — Stub handlers as a Potemkin API

**Setup:** Engagement client in Phase 0 Discovery reviews the API + UI. They see meter CRUD, reports list, tenant settings — all returning 200/201 with reasonable JSON.

**Exploit:** Not a security exploit per se; an HONESTY exploit. The API surface is real (RBAC enforced, auth required), but most endpoints are mocked. `GET /reports/:id/download` returns the literal bytes `%PDF-1.4\n%placeholder\n`. `POST /meters` says 201 Created and persists nothing.

**Impact:** When an engagement consultant demos the system or runs an acceptance test, they get false confidence. In production the meters list will be empty regardless of what was POSTed; the report PDF download will be a 30-byte placeholder. The Italian-flagship claim does not survive a 5-minute hands-on demo with persistence checking.

**Fix:** This is a Sprint S6/S7 priority above Pack-catalogue closure. Wire each placeholder to the repo. Until then, return 501 with `Plan §X.Y not yet implemented`.

### 1.4 Defensive posture — what's done well

- **JWT signing-method whitelist** (`auth.go:179, 236`): `jwt.WithValidMethods([]string{"HS256"})` blocks alg=none / RS256-confusion class.
- **Bcrypt for credentials** (`auth.go:136`).
- **Constant-time comparisons** for CSRF and pulse signature (`security.go:130`, `pulse.go:118`).
- **CSRF double-submit + Bearer bypass** (`security.go:122`): correct pattern.
- **HSTS + CSP + Permissions-Policy + X-Frame-DENY in production** (`security.go:27-58`).
- **Production boot refuses default JWT secret + `sslmode=disable`** (`config.go:175-194`).
- **Distroless non-root image with digest-pinned base** (`backend/Dockerfile:35-37`).
- **K8s pod security context**: `runAsNonRoot`, `readOnlyRootFilesystem`, `allowPrivilegeEscalation: false`.
- **Terraform state encrypted with KMS + DynamoDB lock** (`terraform/versions.tf`).
- **RDS storage_encrypted, publicly_accessible=false, deletion_protection in prod, multi-AZ in prod** (`terraform/main.tf:78-99`).
- **All SQL through parameterised placeholders** (`pgx`, no string concatenation).
- **Bulk inserts via `pgx.CopyFrom`** (efficient + parameterised).
- **`html/template` not `text/template`** for HTML rendering.
- **Brute-force lockout tracker** (`auth_lockout.go`).
- **Audit middleware writes per state-changing request** (read-skip is the gap).

### 1.5 Final verdict (this pass)

**Would NOT pass an enterprise security audit today.** The cryptographic primitives, supply-chain hardening, IaC posture, container security, and CSRF/CSP/HSTS hygiene are textbook-correct — this is real, not cargo-culted. But three CRITICAL bugs (cross-tenant write via missing claim, cross-tenant DPO via URL param, dev-fallback in non-production environments) plus one HIGH (stub handlers presenting as functional) would each be a P0 blocker in any external pen-test report.

**Disposition: "sophisticated security architecture with critical implementation gaps."** The doctrine and ceremony around supply-chain integrity (Cosign, manifest-lock, required-packs enforcement) is documentation-only — the actual code does NOT verify what the docs claim.

**Ship-blocking work required:**

1. Tenant-scoping fix on every handler (`tenant_id` from JWT, never default).
2. DPO cross-tenant gate.
3. Replace stub handlers with real persistence (or 501).
4. Implement OR retract the Cosign/Pack-loader claims.
5. Tighten dev-fallback to `APP_ENV == "development"` exact match.

Until those land, the project is best characterised as **"audit-grade architecture, demo-grade implementation."** The bones are good; the muscle is not yet attached on the data-plane half.

## §2 — Backend Go Code Audit

> **Pass owner:** Senior Go reviewer doing a code-buy-out review.
> **Verdict (this pass):** "Polished demo with deferred integration." Pack code is real and well-engineered; production HTTP path still runs the legacy implementation.

### 2.1 Structural overview

Backend is `cmd/server/main.go` + `internal/{api,config,domain,handlers,jobs,metrics,models,observability,packs,repository,resilience,security,services}` + `packs/{factor,region,report}`. 78 non-test `.go` files (~6 KLoC in critical paths), 45 test files. Layout follows clean-architecture intent: `domain/` holds Pack-contract interfaces (`reporting.Builder`, `emissions.FactorSource`, `region.RegionProfile`); `packs/` holds concrete implementations; `services/` is the orchestration tier; `handlers/` is the HTTP edge; `repository/` is the data tier with `tx.go` exposing `InTxAsTenant` for RLS-scoped transactions. Dep stack mainstream: pgx v5.9.2, Fiber 2.52.12, golang-jwt v5, asynq, OTel 1.43, gobreaker v2, zap. Go 1.26. The `internal/packs/` package defines a Pack manifest schema + `ValidateBasic()` but does **not** implement a loader or Cosign verifier. The `services/pack_registry.go` is a passive lookup over package-level `Pack` globals from the 12 shipped Packs. The legacy `services/report_generator.go` (525 LoC) is the **actual** dispatcher used by HTTP `/reports` — the new Pack registry is **not** wired into it.
